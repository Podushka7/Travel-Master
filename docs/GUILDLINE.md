# Code Style Guideline (Стандарт кодирования проекта Travel Master)

Все разработчики команды (Вася и Таня) обязаны строго соблюдать данные правила при написании кода. Тестировщик (Паша) проверяет код перед слиянием веток.

## 1. Правила именования элементов (Naming)
*   **Названия классов:** Пишутся в стиле CamelCase с большой буквы. 
    *   *Пример из нашего кода:* `Tour`, `TourDatabase`, `MainWindow`.
*   **Названия методов и функций:** Пишутся в стиле camelCase с маленькой буквы.
    *   *Пример из нашего кода:* `append()`, `totalCostByCountry()`, `loadActiveTour()`.
*   **Префиксы слотов Qt:** Автоматические слоты от Qt Designer должны начинаться с `on_` (например: `on_actionOpen_triggered()`, `on_browserItemClicked()`). Вручную созданные слоты пишутся без `on_` в стиле camelCase (например: `saveButton_clicked()`).
*   **Переменные-члены класса (поля private):** Должны начинаться с префикса `m_` для отличия их от локальных переменных (исключение — класс-структура `Tour`, где все поля public).
    *   *Пример из нашего кода:* `m_map`, `m_isModified`, `m_nextId`.

## 2. Структура и организация файлов
*   Заголовочные файлы (`.h`) должны быть защищены от повторного включения классическими директивами `#ifndef / #define / #endif`.
*   Имена файлов должны строго соответствовать имени класса, написанному маленькими буквами: `tour.h / tour.cpp`, `tourdatabase.h / tourdatabase.cpp`.

## 3. Форматирование текста кода
*   **Фигурные скобки `{ }`:** 
    *   Для заголовков функций и классов фигурная скобка открывается на **новой строке**:
        ```cpp
        int TourDatabase::count() const
        {
            return m_map.size();
        }
        ```
    *   Для управляющих конструкций (`if`, `while`, `for`) открывающая скобка ставится **на той же строке**:
        ```cpp
        if (fileName.isEmpty()) {
            return;
        }
        ```
*   **Отступы:** Стандартный отступ кода внутри блоков равен 4 пробелам (настройка табуляции в Qt Creator).

## 4. Ограничение сложности и правила комментирования
*   **Запрет на "магические числа":** Диапазоны валидаторов и цен должны настраиваться программно (например, `ui->spinCost->setRange(3000, 100000)`).
*   **Комментарии:** Короткие однострочные комментарии со знаком `//` ставятся перед сложными блоками кода или внутри объявлений классов для пояснения назначения переменных (например: `mutable bool m_isModified; // Флаг изменений`). Очевидные вещи комментировать запрещено.

## 5. Культура работы с Git и коммитами
Каждое сообщение к коммиту должно содержать префикс, указывающий на характер изменений:
*   `feat:` — добавление новой логики (например: `feat: add sale button for Egypt`)
*   `fix:` — исправление багов (например: `fix: fix row index in delete tour`)
*   `docs:` — изменение документации (например: `docs: create readme and guideline`)
