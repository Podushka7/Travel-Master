# Устав проекта: Информационная система «Travel Master»

## 1. Термины и сокращения
*   **ИС** — информационная система.
*   **ПП** — программный продукт.
*   **MVP** (Minimum Viable Product) — минимально жизнеспособный продукт (первая рабочая версия программы).
*   **QA** (Quality Assurance) — обеспечение качества (тестирование).
*   **НСИ** — нормативно-справочная информация (справочники стран, видов отдыха и мест).

## 2. Команда проекта и распределение Scrum-ролей
Все участники команды являются инженерами, принимают участие в написании кода и выполняют ключевые обязанности в рамках ЖЦ проекта:

*   **Богдан Ахлиманов** — Менеджер проекта / **Scrum Master**. Отвечает за общую координацию, устранение препятствий, ведение Scrum-доски и соблюдение регламентов.
*   **Вася Питонов** — Ведущий разработчик / **Product Owner**. Отвечает за архитектуру ядра, техническую стратегию, управление бэклогом продукта и приоритеты функций (контейнер `QMap`, сериализация).
*   **Паша Багов** — Тестировщик / **Developer (QA)**. Отвечает за формирование тест-кейсов, ручное и автоматизированное тестирование, проверку граничных условий UI и аудит кода на соответствие Guideline.
*   **Таня Сишник** — Разработчик / **Developer**. Отвечает за верстку графического интерфейса (`mainwindow.ui`), проектирование форм ввода, связывание сигналов и слотов, точечное обновление виджетов.

## 3. Предпосылки инициации проекта
Малые и средние туристические агентства сталкиваются с поляризацией рынка ПО: им доступны либо тяжелые облачные CRM-системы с высокой абонентской платой и риском утечки клиентской базы, либо неспециализированные таблицы Excel, не имеющие жесткой бизнес-валидации полей. Основными предпосылками разработки являются:
1.  Необходимость обеспечения 100% конфиденциальности данных клиентов за счет локального хранения.
2.  Высокий риск совершения сотрудниками ошибок при ручном вводе финансовых данных и параметров визового контроля.
3.  Потребность в легковесном, переносимом (portable) решении с мгновенным откликом интерфейса.

## 4. Цели проекта
1.  Разработать надежный инструмент локального учета туров с жесткой валидацией вводимых типов данных на уровне UI.
2.  Исключить риски человеческого фактора при расчете стоимости и проверке визовых требований для путевок.
3.  Обеспечить автономность работы менеджеров без обязательного подключения к сети Интернет.

## 5. Границы проекта (Высокоуровневое содержание)
Программа реализует строго определенный набор функций для автоматизации работы:
*   **Управление данными (CRUD):** Создание, просмотр, обновление и удаление карточек туров в оперативной памяти.
*   **Параметры тура:** Вид отдыха, Страна (Россия, Абхазия, Турция, Египет), Место отдыха, Стоимость, Длительность, Виза.
*   **Файловые архивы:** Сохранение текущей базы в кастомный бинарный формат `.tdb` и пошаговая загрузка первых $N$-записей из внешних архивов.
*   **Фильтрация:** Автоматический отбор и вывод записей в браузере интерфейса по выбранной пользователем стране.

## 6. Требования к продукту и проекту
*   **Интерфейс:** Реализация на нативных компонентах `Qt Widgets` без размытия на экранах с высоким разрешением (High-DPI).
*   **Валидация:** Поле «Место отдыха» должно проверяться регулярным выражением (первая заглавная, только буквы, длина до 50 символов). Поля цены и дней должны иметь жесткие программные лимиты.
*   **Надежность:** Флаг изменений базы `m_isModified` должен сбрасываться только после успешного завершения записи потока `QDataStream` на диск.

## 7. Допущения и ограничения проекта
*   **Ограничение по времени:** Продукт (MVP) должен быть полностью разработан, протестирован и зафиксирован в репозитории до окончания текущего учебного семестра.
*   **Ограничение по стеку:** Разработка ведется строго на языке C++11 с использованием фреймворка Qt 5/6 и системы сборки QMake.
*   **Допущение по команде:** Состав проектной команды (4 человека) и распределение ролей остаются неизменными на протяжении всех спринтов.

## 8. Критерии успешности проекта
*   **По содержанию:** Приложение успешно выполняет все операции CRUD, автоматические скидки на туры в Египет и частичный импорт без коллизий ID.
*   **По качеству:** Исходный код Васи и Тани успешно проходит аудит Паши на соответствие `GUIDELINE.md` перед слиянием веток.
*   **По управлению:** Все задачи спринтов зафиксированы в таск-трекере, а итоговый код отправлен в приватный репозиторий GitHub.

## 9. Регламент работы по методологии Scrum
1.  **Длина спринта:** Устанавливается фиксированный шаг итерации длительностью в **1 неделю**.
2.  **Планирование спринта:** Проводится Scrum-мастером (Богданом) совместно с командой в начале итерации. Задачи декомпозируются на подзадачи объемом не более 16 часов.
3.  **Приемка задач:** Перевод карточки задачи в статус «Done» осуществляется PM только после верификации функционала QA-специалистом (Пашей) и фиксации коммита в Git.
